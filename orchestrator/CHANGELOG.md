# Orchestrator workflow changelog

`WORKFLOW_VERSION` (in `orchestrator/workflow.py`) is the version of the
LangGraph `@entrypoint` **body**. It is bumped on INCOMPATIBLE changes to that
body — reordered/removed tasks, new required tasks, or changed control flow that
a half-finished checkpoint can't safely resume into. Pure additive changes (a new
optional gate, a new trailing task that legacy checkpoints simply haven't reached)
don't strictly require a bump.

On every resume, the version stored at run creation is compared against the live
constant; a mismatch refuses the resume with a clear error rather than risking a
confusing deserialization failure mid-run.

This file is the human history behind each bump. The deep design notes for each
phase live in `.misc_notes/Done/`.

| Version | Phase | Change |
|---|---|---|
| 2.7.0 (unchanged) | 78b | **TDD-aware decompose** (second slice of Phase 78 — TDD cost/efficiency; independent of the Phase 77 evidence layer). Under TDD a separate test-author writes each task's FAILING tests BEFORE implementation, so a standalone "write the unit tests" task is redundant and harmful — it runs after the code exists, can't go red, and degrades to classic (the Phase 75 dogfood split the feature into "implement cashToClose" + "write the unit tests" and had to be re-planned by hand). When `config.tdd`, a note is now composed into the decomposer's USER MESSAGE — `_build_user_message(plan, max_tasks, tdd=True)` appends `_TDD_DECOMPOSE_NOTE` ("a separate test-author writes the failing tests for each task before it is implemented … do NOT emit any standalone 'write tests' / 'add tests' / 'unit tests' task; every task is behaviour-only, with acceptance criteria the test-author will turn into tests"). `config.tdd` is threaded `_plan_and_approve` → `decompose_task` → `decompose` as a defaulted `tdd` param. It is a prompt-level instruction, NOT a testability judgement — the decomposer still reads only `plan_text` with no repo access (the Phase 73 (a) trap stays avoided); it only forbids test-writing TASKS, which is safe to state unconditionally under TDD. The note rides the user message (conditional on `config.tdd`), not the static `decompose.md`, so with TDD off the message, the behaviour, and the `DecompositionResult` are byte-for-byte unchanged. An additive defaulted param on a checkpointed `@task` (a completed `decompose_task` replays its cached result by position, not by re-invoking with stored args) with no result-shape or task-graph change → **`WORKFLOW_VERSION` stays 2.7.0**. |
| 2.7.0 (unchanged) | 78a | **Coverage critic on Haiku** (first slice of Phase 78 — TDD cost/efficiency; highest $/effort). The per-task coverage critic is a read-only test-*meaningfulness* judge (does a test pin behaviour, or is it vacuous/tautological/shape-only?) — a classification task Haiku handles well, and with TDD on by default (Phase 75) it runs on EVERY testable task (~$0.20/task on Sonnet in the dogfood). The bundled `prompts/coverage-critic.md` now pins `model: haiku` in its frontmatter, which `_coverage_critic_model` already reads (`config.resolved_model(load_prompt_frontmatter("coverage-critic").model)`): silent before → fell back to `default_model` (Sonnet); now it resolves to the full Haiku id. The lever is the per-token PRICE, not the token count — the prompt body/scope is untouched, since those tokens are legitimate review work (capping output would weaken the judgement, and fail-open means the real risk of a weaker model is the opposite, weak tests waved through). It is the bundled DEFAULT, not a code hardcode — a project can still point the critic at any model via its own `.orchestrator/prompts/coverage-critic.md`. The frontmatter is stripped from the system-prompt body as always. No code, schema, or task-graph change → **`WORKFLOW_VERSION` stays 2.7.0**. |
| 2.7.0 (unchanged) | 77d | **QA scoping + test-plan under TDD** (fourth/final slice of the Phase 77 evidence layer; independent of 77a–c). Two boundary fixes from the Phase 75 dogfood review. (1) **`qa.md` is now strictly the QA agent's own record.** `QaResult` gains an optional `review` field — the agent's account of what it reviewed and ran (each static check + checklist item with its ✓ PASS / ✗ FAIL), emitted via the QA agent's `emit_qa_result` tool (new `review` field + a `## When done` section in `prompts/qa.md` that step 6 already referenced but didn't exist). `write_qa` renders it as a `## Checks performed` section above any `## Failures`, replacing the bare one-line `# QA Result: PASS`. The red-green results never bleed in — they stay in the per-task `test-author/` + `impl/` folders — so a reader never has to disambiguate the two records. (2) **The implementer's `test-plan.md` is dropped when `config.tdd` is on** (`write_summary(..., tdd=config.tdd)`): under TDD it was both redundant and misleading (the Phase 75 run claimed "8 tests" for 9 and listed untested speculation), and the executed evidence supersedes it; the gate is strictly `tdd`, so with TDD off it is still written (a manual checklist earns its place where there is no automated suite). `QaResult.review` is a pure-additive optional field (no `schema_version` bump) and the rest is artifact-shaping only — **no `@task`, no checkpointed-shape change → the task graph is byte-for-byte unchanged and `WORKFLOW_VERSION` stays 2.7.0**. The `{summary, test_plan}` MCP/PR-body shape is untouched; only the on-disk `test-plan.md` artifact is gated. |
| 2.7.0 (unchanged) | 77c | **Impl attempt evidence** (third slice of the Phase 77 TDD-evidence layer; the FIRST consumer of BOTH 77a seams). For each implement attempt of a TESTABLE TDD task the station now writes `task-NN-<id>/impl/attempt-N/` — `test-results.md` (the COMPLETE test run after this attempt via 77a's `StepResult.full_output`; every gate that produced a captured runner log, so the synthetic diff-gate and the LLM `builtin:qa` gate fall out and what remains is the actual suite run — proof the WHOLE suite ran, not just the failing delta) and `snapshot-hash.md` (a re-hash of `test_paths` this attempt vs the frozen 77b baseline → explicit MATCH ✓ / MISMATCH ✗; a MISMATCH is the evidence of *why* the diff-gate failed the attempt). Wired through 77a's `run_retry_block(on_attempt=…)` hook, which fires ONCE per attempt after the gates resolve for EVERY attempt — crucially the GREEN one — threaded as an optional read-only observer through `_run_build_step` / `_run_task_build_step`; `_impl_attempt_recorder` closes over the task's baseline and re-hashes with the same pure `_hash_test_paths` the diff-gate uses. `attempt` is run_retry_block's own 1-based counter (continuous across a growable budget); on `attempt <= 1` the impl/ folder is cleared, so a fresh build — notably each autonomous re-author round (Phase 76), which re-freezes a NEW suite and restarts the count — leaves only the attempts against the suite that ultimately ran, never stale higher-numbered folders from a replaced suite. The recorder is a best-effort side-effect in the entrypoint frame (like the 77b writer), so it replays cleanly on resume; **no new `@task` and no checkpointed-shape change → the task graph is byte-for-byte unchanged and `WORKFLOW_VERSION` stays 2.7.0**. The classic / non-TDD / untestable path passes `on_attempt=None` and writes nothing here, so those runs are byte-for-byte identical. |
| 2.6.0 → 2.7.0 | 77b | **Test-author evidence folder** (second slice of the Phase 77 TDD-evidence layer). The flat `test-author-<id>.md` is replaced by a per-task `task-NN-<id>/test-author/` folder, written ONCE after the authoring process converges (critic satisfied + any re-author done): for a TESTABLE task it copies the FINAL accepted `test_paths` file(s) verbatim (any language/layout, preserving relative paths), plus `results-test-run.md` (the COMPLETE RED run via 77a's `full_output`, proof the suite actually ran red — not just the failing delta), `test-snapshot-hash.md` (the deterministic `_hash_test_paths` baseline the diff-gate freezes), and `summary.md` (testable verdict, coverage-critic verdict(s), # re-author rounds, red-review outcome). EVERY TDD task still gets a folder (Phase 73 surfacing): an untestable / degraded task gets `summary.md` only. `TestAuthorResult` gains a `full_run` field (the complete RED run; `red_output` stays the failures-only summary that feeds the implementer's first attempt + the red-review display) — a checkpointed shape change on a serde-allowlisted model, so a half-finished 2.6.0 TDD run can't resume across the upgrade; hence the bump. Landmine fixed: `_hash_test_paths` (and the new copier) now resolve `test_paths` through `paths.iter_test_files`, which excludes the orchestrator's own `.orchestrator/` workspace — without it the verbatim copies written there would be swept back into the freeze hash by `**` and break the diff-gate. The post-branch task graph is unchanged (the writer is a side-effect after the authored verdict, like its predecessor) and TDD-off runs are byte-for-byte identical. |
| 2.5.0 → 2.6.0 | 77a | **Evidence foundations** (generic plumbing for the Phase 77 TDD-evidence layer). Two additive, behaviour-preserving seams, both inert until a consumer is wired in 77c: (1) `StepResult` gains `full_output` — the COMPLETE, untruncated runner log (both streams), so the evidence layer can persist the whole RED/attempt run; `detail` stays the short tail/failure report it always was. `_run_script_sync` now fills it on the success and gate-fail paths. (2) `run_retry_block` gains an `on_attempt(attempt, passed, gate_results)` hook that fires ONCE per attempt after the gates resolve, for EVERY attempt — crucially the passing (GREEN) one, which `on_gate_failed` never sees and the old `for…else` returned on immediately; it fires before `on_gate_failed` so an attempt's evidence is recorded before any human decision. The post-branch task graph is unchanged and with no hook/consumer the behaviour is byte-for-byte identical — but a checkpointed `StepResult` shape changed, so a half-finished 2.5.0 TDD run can't resume across the upgrade and needs a fresh start; hence the bump. |
| 2.4.0 → 2.5.0 | 76 | **Autonomous-mode TDD** (re-enable the guard). `tdd = true` + `fully_autonomous = true` is no longer refused at load; instead the per-task station runs a distinct `_run_autonomous_tdd_task` when both are on. With no human red-review/re-author guard, two pieces of machinery stand in: (1) **red-confirm is a HARD gate** — a born-green / non-green-baseline / no-script-gate verdict (`TestAuthorResult.degrade_kind` ∈ the red-confirm failures) aborts the task (`BuildFailed` → `status="failed"`) instead of silently degrading; a genuinely-untestable verdict (the author's own judgement) still degrades to classic implement→qa + a manual check; (2) **bounded re-author** — the implement build runs with a BOUNDED budget and `on_exhausted="abort"`, and an exhausted build automatically re-authors the (possibly wrong) tests, capped at `tdd_autonomous_reauthor_max` (default 2), then aborts with a clear reason rather than looping a wrong frozen test to the `autonomous_max_*` safety ceiling. New reachable control flow + per-round `attempt` discriminators on `test_author_task`/`critic_task` → a half-finished 2.4.0 checkpoint can't safely resume into it. The supervised / non-TDD per-task graph is byte-for-byte unchanged; `degrade_kind` is an additive optional field, so no serde-allowlist change. `_author_with_review` now returns `(verdict, attempt)` and seeds from a caller-supplied `feedback`/`attempt` so the autonomous cycle can drive repeated rounds with a monotonic counter. |
| 2.3.0 → 2.4.0 | 74 | Per-task **coverage critic** (test-meaningfulness backstop). `_author_with_review` now runs a new checkpointed `critic_task` (a separate read-only `coverage-critic` agent) on the authored tests; a `meaningful=False` verdict re-authors via the existing escape (bounded by `tdd_critic_max_attempts`), and if still weak after the budget the run proceeds and records the concern in `manual_checks` (never wedges). A new `@task` in the body's graph → a half-finished 2.3.0 checkpoint can't safely resume into it. Gated on `config.tdd_coverage_critic` (default on); `CoverageCriticResult` on the serde allowlist. Generic (a separate agent, not an augmentation of the project-specific `qa.md`). |
| 2.2.0 → 2.3.0 | 72b | Supervised **red-review pause + re-author escape**. The per-task station now runs `_author_with_review`: after the green→red confirm it emits a new `red_review` interrupt (default-on via `tdd_red_review`) for a human to review the failing tests — `yes` implements, feedback re-authors the tests (re-running `test_author_task`, which gained `feedback`/`attempt`), `abort` stops the run (`BuildFailed` → `status="failed"`). New interrupt + re-author control flow the body can reach → a half-finished 2.2.0 checkpoint can't safely resume into it. Suppressed when `tdd_red_review=false` (or `fully_autonomous`). Surfaced through the existing `approve_plan`/`resume` MCP contract (no new tool). |
| 2.1.0 → 2.2.0 | 72b | `Task.acceptance_criteria` became **required** (was optional, `default=None`) so every vertical slice carries the observable-behaviour spec the test-author writes tests against; `DecompositionResult.schema_version` 1 → 2. The decomposer prompt already demanded it (Phase 70); this aligns the model. Because `DecompositionResult` is checkpointed, a pre-2.2.0 checkpoint can hold a task with no criteria that no longer deserializes — the version gate refuses that resume cleanly rather than risking a mid-run `ValidationError`. (Earlier 72b slices — `test_author_path`, frontmatter model/tools, RED→implementer + artifact — were additive and needed no bump.) |
| 2.0.0 → 2.1.0 | 72 | The per-task station gained a run-once, checkpointed `test_author_task` before each task's implement loop, gated on `config.tdd` (TDD red-green). When on: a separate test-author agent writes failing tests once, the body confirms the green→red transition, and a `builtin:diff-gate` (an injected gate callable, ordered first) freezes the tests so the implementer can't edit them. A new `@task` in the body's graph → a half-finished 2.0.0 checkpoint can't safely resume into it. With `tdd` off (the default) the task is never called and the post-branch graph is identical to 2.0.0, but the body now *can* emit it, so the bump. New `TestAuthorResult` on the serde allowlist; graceful-degrade (untestable / born-green / no-script-gate → classic implement→qa). |
| 1.11.0 → 2.0.0 | 68b | The declarative-pipeline cutover. The body now dispatches the post-branch stages from `config.pipeline` (the v2 `flow`) instead of the hard-coded spine + v1 `[[steps.work]]` seam, and the manifest-hash resume gate became a pipeline-hash gate (`record_pipeline_hash_task`). An incompatible body+gate change → old 1.x checkpoints cannot resume. (Retroactively recorded — the constant moved to 2.0.0 in Phase 68b; this row documents that jump.) |
| 1.0.0 → 1.1.0 | 33 | Body gained the manifest-hash gate and five `run_seam()` insertion points, plus new tasks (`record_manifest_hash_task` and the per-step tasks built by `_make_script_task` / `_make_ai_agent_task`). A body change → the bump makes any pre-33 checkpoint refuse to resume rather than resume into a changed task graph. (Phase 33's later refinements — per-id step task names, `after_qa` firing only on PASS, `approval_gate` abort — all landed before 1.1.0 shipped, so they fold into this version.) |
| 1.1.0 → 1.2.0 | 41 | `docs_task` is now a permanent spine task, inserted after the `before_commit` seam and before `commit_task`. A new required task is a control-flow change → the bump refuses resume of any pre-41 run rather than resuming into a task graph that lacks the docs step. |
| 1.2.0 → 1.3.0 | 42 | The hand-written impl↔QA retry loop is replaced by the generic retry engine (`retry_block.run_retry_block`); implementation is now a generic producer `@task` (no `ImplementationResult`); a new `summarize_task` runs after the block to derive the commit/PR summary + test_plan from the diff. New/removed task names + changed control flow = a body change. |
| 1.3.0 → 1.4.0 | 46 | The impl↔QA loop is no longer inline in the body — it runs through `run_seam("after_branch")` as a declarative `build` step (synthesized by default; overridable in `orchestrator.toml`).
